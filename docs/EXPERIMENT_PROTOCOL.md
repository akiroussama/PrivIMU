# Experiment Protocol

## Research question

Can IMU traces that do not contain direct personal identifiers still reveal the identity of an enrolled user?

## Dataset

PrivIMU uses MotionSense, a public smartphone sensor dataset. The experiment uses the `A_DeviceMotion_data` folder because it contains device-motion signals derived from accelerometer and gyroscope sources.

## Target variable

The target is the **subject ID**. This is a closed-set re-identification experiment over 24 enrolled subjects.

## Input channels

Default channels:

```text
rotationRate.x, rotationRate.y, rotationRate.z,
userAcceleration.x, userAcceleration.y, userAcceleration.z
```

This keeps the story focused on gyroscope + accelerometer behavior.

## Preprocessing

1. Read each MotionSense CSV file.
2. Select six IMU channels.
3. Segment into 1-second windows at 50 Hz.
4. Use 50% overlap.
5. Apply z-score normalization per window and per channel.
6. Extract 60 features: 10 descriptors × 6 channels.

## Models

### Random Forest

The RF baseline is the primary model because it is fast, interpretable, CPU-friendly, and easy to defend during a viva.

### 1D-CNN

The CNN is an extension for raw-window learning. If CPU training is too slow, the project remains valid with RF only.

## Split strategy

The default training script uses group-aware evaluation by `activity_trial`. This prevents windows from the same activity/trial group from being mixed across train and test folds.

Available group keys:

```text
activity_trial   example: wlk_03
trial_instance   example: sub07_wlk_03
```

Final results should mention the exact `split_strategy` stored in `reports/metrics.json`.

## Metrics

- Top-1 accuracy
- Top-3 accuracy
- F1-macro
- Confusion matrix
- Per-subject F1
- Latency per window
- Privacy entropy leakage: `ΔH = log2(N) − H(posterior)`

## Anti-hallucination rule

No result is valid unless it is generated by code and appears in `reports/metrics.json`.
